OUTTACOUCH

1) Project Overview

OUTTACOUCH is an event-first social connection platform designed to translate online discovery into offline relationships.
The core problem is that traditional social apps prioritize engagement loops (feeds, likes, comments) instead of facilitating real-world interactions.
OUTTACOUCH exists to make event attendance the primary signal of identity and trust, so connections form around shared experiences rather than superficial profile activity.

Why this project exists:
- People want meaningful connections but lack a reliable way to meet others around common interests.
- Hosts need a streamlined way to list events, manage attendance, and validate entry.
- Existing social graphs are noisy and do not reflect real-world context.

Real-world use cases:
- A user discovers a nearby music event, commits to attend, and later connects with other attendees.
- A host creates an event (free or paid), manages capacity, and validates entry with QR scans.
- A user builds a calendar-first profile that shows attended and hosted events as their social identity.
- A user invites phone contacts via SMS or WhatsApp using a referral link; when the contact signs up, they are automatically connected.

Target users:
- Attendees who want to meet new people through shared events.
- Hosts who need tools for event publishing, attendee management, and check-ins.
- Communities that prefer real-world trust signals over follower counts.

Key product promises:
- Relevance: discovery is ranked by location, timing, and interest match.
- Trust: connections are anchored in shared attendance.
- Action: the UX pushes users from browsing to committing and showing up.


2) System Design (High-Level + Low-Level)

High-level architecture is a single Next.js 15 application that serves both the frontend and the backend API.
The frontend uses the App Router with route groups: app/(auth) for authentication and onboarding flows, and app/(main) for the protected core product (explore, events, chat, profile, settings, notifications).
The backend is implemented as Next.js API routes in app/api, and a Socket.io server bootstrapped through pages/api/socketio.ts for realtime chat.
Core services: PostgreSQL via Prisma 7 ORM, Supabase Storage (image uploads), Twilio Verify (phone OTP), Twilio SMS/WhatsApp (invitations), and SMTP/Nodemailer (email OTP and notifications).
Payments: no online payment processing. Paid events show a "contact the host directly" message. Tickets are still tracked and QR-validated for entry.

Low-level design splits features into modules: authentication, event discovery and management, ticketing, connections, chat, memories, notifications, referrals/contacts, and storage.
Each module has API endpoints that map directly to UI screens and actions. Data access flows through Prisma queries with explicit ownership checks at the endpoint level.
Realtime messaging uses Socket.io rooms keyed by connectionId.
Notification dispatch runs as a Vercel Cron job every 5 minutes, hitting GET /api/notifications/dispatch secured by CRON_SECRET.

Data flow in a typical request:
Client action -> App Router page -> fetch to app/api endpoint -> JWT validation -> Prisma query -> JSON response -> UI state update.

Scalability considerations:
- API routes are stateless and scale horizontally.
- Prisma uses pooled DB connections (PgBouncer via Supabase); DIRECT_URL is used for migrations to bypass the pooler.
- Event feed is capped to 50 results and scored in memory.
- Socket.io is single-node and would require a Redis adapter for multi-node scaling.
- Supabase Storage is scalable for media.

Trade-offs:
- Monolithic Next.js app simplifies development and deployment but couples API and UI and requires extra work for realtime scaling.
- Event-first connections improve relevance and safety but reduce the speed of discovery compared to follower-based models.
- No online payments keeps the system simpler and avoids PCI scope, but limits monetization.

System overview (text diagram):
Client (Next.js App Router UI)
  -> API routes (app/api) for auth, events, tickets, chat, connections, notifications, referrals
  -> Prisma 7 ORM -> PostgreSQL (Supabase, pooled via PgBouncer)
  -> Supabase Storage for images
  -> Twilio Verify for OTP / Twilio SMS+WhatsApp for invitations
  -> Nodemailer/SMTP for email OTP and notifications
Realtime:
  Client <-> Socket.io server (pages/api/socketio.ts) <-> globalThis.io emitToRoom (lib/socketServer.ts)
Cron:
  Vercel Cron -> GET /api/notifications/dispatch (every 5 minutes)


3) Tech Stack and Design Choices

Frontend:
- Next.js 15 App Router + React 18: unified routing, server/client component split, API colocation.
- TypeScript: type-safe props, API payloads, and shared types across client and server.
- Tailwind CSS with custom design tokens: ink, parchment, ocean, accent, bg, subtle, border.
- Framer Motion: swipe and card interactions for event discovery.
- Leaflet (react-leaflet): interactive maps for event location and picker.
- Recharts: host analytics charts.
- react-hook-form: efficient form state and validation.
- Zustand: minimal global state (authStore, uiStore).

Backend:
- Next.js API routes: colocated API for rapid development and deployment.
- NextAuth v4 Credentials + JWT: session management via JWT, getToken in API routes, getServerSession in some routes.
- Prisma 7 with PrismaPg adapter: typed queries, migrations (using DIRECT_URL to bypass pooler), and transactional integrity.
- PostgreSQL (Supabase): relational model fits events, tickets, and connections.
- Socket.io: realtime chat, typing indicators, read receipts.

Integrations:
- Supabase Storage (admin key, server-side): scalable public media storage.
- Twilio Verify: phone OTP for signup and password reset.
- Twilio SMS + WhatsApp: referral invitation delivery.
- Nodemailer/SMTP: email OTP and notification delivery.
- Web Contact Picker API (navigator.contacts): device contact sync on supported mobile browsers.

Design choices:
- Monolithic Next.js app for faster iteration and simpler deployment.
- App Router for core UI; pages/api used only to bootstrap Socket.io (App Router does not support persistent connections).
- JWT claims (profileComplete, isDeactivated) enforce onboarding and access at the middleware level.
- Server-side Supabase uploads avoid exposing storage service credentials to the client.
- No online payments: removes PCI scope and Stripe dependency; paid events route hosts and attendees to settle directly.
- Vercel Cron for notification dispatch: no separate worker process needed.


4) Database Design

Schema uses a relational model in PostgreSQL managed by Prisma 7.
The datasource URL is a PgBouncer pooled connection string (DATABASE_URL) for runtime, and a direct connection string (DIRECT_URL) in prisma.config.ts for migrations.

Models:

User
  - id, email, phone, passwordHash, displayName, bio, profilePhotoUrl
  - city, lat, lng (location for discovery scoring)
  - preferences (string array of interest categories)
  - profileComplete, isDeactivated
  - visibility (Visibility enum), approvalMode (ApprovalMode enum)
  - Relations: hosts Events, EventAttendees, Tickets, Connections (sent/received), Messages, Notifications, NotificationSchedules, OtpTokens, Memories, EventSwipes, ContactImports, ContactInvitations, ReferralLinks

Event
  - id, title, description, category, location (string address), lat, lng
  - startAt, endAt, maxAttendees, currentAttendees
  - isFree, price (numeric), currency
  - status (EventStatus enum), approvalMode
  - hostId -> User
  - Relations: EventImages, EventAttendees, NotificationSchedules, Tickets, shared Connections, Memories, EventSwipes

EventImage
  - id, eventId, url, order, isCover

EventAttendee
  - id, eventId, userId, status (AttendeeStatus enum)
  - Unique index on (eventId, userId)

Ticket
  - id, eventId, userId, eventAttendeeId (optional link)
  - paymentStatus (PaymentStatus enum)
  - qrCode (unique), validatedAt
  - quantity, amount, currency

Connection
  - id, user1Id, user2Id, status (ConnectionStatus enum)
  - sharedEventId (optional, for event-based connections)
  - requestedAt, acceptedAt

Message
  - id, connectionId, senderId, content, type (MessageType enum)
  - sentAt, readAt

Notification
  - id, userId, title, body, link
  - createdAt, readAt (null = unread)

NotificationSchedule
  - id, userId, eventId (optional), type, sendAt
  - sent (boolean), createdAt
  - Unique index on (userId, eventId, type, sendAt)

OtpToken
  - id, contact, otp (hashed for email), purpose (signup/reset)
  - verifiedAt, usedAt, expiresAt, userId (optional)
  - Index on (contact, purpose, usedAt)

Memory
  - id, userId, eventId, imageUrl, caption, createdAt

EventSwipe
  - id, userId, eventId, direction (like/pass), createdAt

ContactImport
  - id, userId, name, phone, status (pending/invited/registered)
  - invitedAt, registeredAt, registeredUserId
  - Unique on (userId, phone)

ContactInvitation
  - id, fromUserId, toPhone, channel (sms/whatsapp)
  - status (pending/sent/clicked/registered)
  - referralCode (unique, used as the join link code)
  - sentAt, clickedAt, registeredUserId

ReferralLink
  - id, code (unique), fromUserId, type (contact/general)
  - clicks, registrations, expiresAt

Enums:
- Visibility: private, connections, public
- ApprovalMode: auto, manual
- EventStatus: upcoming, ongoing, past, cancelled
- AttendeeStatus: committed, attended, missed, waitlisted
- ConnectionStatus: pending, accepted, declined, removed
- MessageType: text, event_card, location
- PaymentStatus: pending, paid, refunded, failed

Indexing:
- Unique on user.email, user.phone
- Unique on ticket.qrCode
- Unique on (eventId, userId) in EventAttendee
- OtpToken indexed on (contact, purpose, usedAt)
- NotificationSchedule unique on (userId, eventId, type, sendAt)
- ContactImport unique on (userId, phone)
- ContactInvitation indexed on referralCode (unique), fromUserId, toPhone
- ReferralLink indexed on code (unique), fromUserId

Data lifecycle:
- Deactivated users keep their row but email/phone are cleared for re-signup safety.
- Event deletion cascades to attendees, tickets, swipes, images, and memories links.
- OTP tokens expire and are marked used/verified as part of each auth flow.
- ContactInvitation progresses: pending -> sent -> clicked -> registered.
- When a referred user registers, a Connection is auto-created with status "accepted" and both users receive a welcome Notification.

Query optimization:
- Event feed caps to 50 rows with computed scoring (distance, preferences, timing, popularity).
- Connection suggestions filter on existing relationships and limit results.
- Notifications limited to 50 by createdAt desc.
- Transactions used for ticket creation and attendance updates to prevent race conditions.
- Prisma select used to avoid over-fetching columns.


5) Backend Deep Dive

API style: REST endpoints under app/api. All request/response bodies are JSON except storage upload (multipart/form-data).

Authentication endpoints:
- POST /api/auth/send-otp { contact, type, purpose } -> { status }
  Sends email OTP (Nodemailer) or phone OTP (Twilio Verify) for signup or reset.
- POST /api/auth/verify-otp { contact, otp, purpose } -> { token }
  Verifies the OTP; returns a signed token for use in the register/reset step.
- POST /api/auth/register { contact, password, token, ref? } -> { status }
  Creates the user, marks OTP used; if ref is a valid ContactInvitation.referralCode, completes referral: marks invitation "registered", updates ContactImport, creates Connection (accepted), sends welcome Notifications to both parties, increments ReferralLink.registrations.
- POST /api/auth/reset-password { contact, password, token } -> { status }

Event endpoints:
- GET /api/events -> scored feed of up to 50 events
- POST /api/events { title, category, location, lat, lng, startAt, endAt, maxAttendees, isFree, price, approvalMode } -> { id }
- GET /api/events/:id -> full detail with host, images, attendee count, analytics (host only)
- PATCH /api/events/:id -> update event (host only)
- DELETE /api/events/:id -> delete event and cascade cleanup (host only)
- POST /api/events/:id/commit -> create EventAttendee in a transaction, increment currentAttendees
- GET /api/events/host -> hosted events for current user
- POST /api/events/swipe { eventId, direction } -> record EventSwipe
- POST/DELETE /api/events/:id/images -> upload or delete event gallery images (Supabase Storage, server-side)

Ticket endpoints:
- GET /api/tickets/me -> user's tickets
- POST /api/tickets { eventId } -> create ticket (status pending; host confirms payment out-of-band)
- GET /api/tickets/:id/qr -> QR code for entry validation
- POST /api/tickets/validate { qr_code, event_id } -> validate QR at entry (host scanner)
- POST /api/tickets/refund { ticketId } -> cancel ticket if within 48h window, set paymentStatus "refunded", decrement capacity

Connection endpoints:
- GET /api/connections -> accepted connections
- GET /api/connections/discover -> users near current user
- GET /api/connections/suggestions -> users with shared event attendance
- GET /api/connections/requests -> pending incoming requests
- POST /api/connections/request/:userId -> send connection request
- GET/DELETE /api/connections/:id -> get or remove connection
- PUT /api/connections/:id/accept -> accept request
- PUT /api/connections/:id/decline -> decline request

Chat endpoints:
- GET /api/chat -> conversation list with last message and unread count
- GET /api/chat/:connectionId -> message history (only connection members)
- POST /api/chat/:connectionId { content, type } -> send message; emits via Socket.io
- PUT /api/chat/:connectionId/read -> mark messages read

Notification endpoints:
- GET /api/notifications -> list of notifications (newest 50)
- PUT /api/notifications/:id/read -> mark single notification read
- PUT /api/notifications/read-all -> mark all read
- GET /api/notifications/dispatch -> Vercel Cron handler (also accepts POST); finds pending NotificationSchedules, sends email/SMS, marks sent
- POST /api/notifications/subscribe -> stub for future push notification device token registration

Referral endpoints:
- GET /api/referrals -> stats + invitations list + referral links for current user
- POST /api/referrals/share { contactIds?, phones?, channel } -> sends SMS/WhatsApp invitation via Twilio; creates ContactInvitation + ReferralLink per contact
- GET /api/referrals/:code -> validates referral code, returns referrer info, tracks click; used by /join page

Contact endpoints:
- GET /api/contacts/sync -> returns ContactImport list + stats (registered/invited/pending)
- POST /api/contacts/sync { contacts: [{name, phone}] } -> upserts ContactImport rows; checks which phones are already registered users

User endpoints:
- GET/PATCH /api/users/me -> get or update current user profile
- PATCH /api/users/me/location { lat, lng, city } -> update location
- PATCH /api/users/me/privacy { visibility } -> update visibility
- GET /api/users/:id -> get public profile (visibility-gated)

Storage endpoint:
- POST /api/storage/upload (multipart) -> uploads to Supabase Storage via admin key, returns public URL

Authorization and security:
- NextAuth JWT validated in all protected API routes via getToken.
- Ownership enforced at endpoint level (event edit/delete checks hostId, chat access checks connection membership, ticket validation checks host role).
- Middleware redirects unauthenticated users to /login, incomplete profiles to /onboarding/location, deactivated users to /deactivated.
- Notification dispatch protected by dual auth: NOTIFICATION_DISPATCH_SECRET header or Vercel CRON_SECRET Bearer token.
- OTP resend cooldown enforced by checking latest OTP timestamp; Twilio Verify adds server-side rate limits.
- Supabase service role key never leaves the server.

Error handling:
- API routes return JSON { error: string } with status codes.
- Common statuses: 400 bad request, 401 unauthorized, 403 forbidden, 404 not found, 409 conflict, 410 gone (deprecated endpoints), 500 server error.
- Referral completion in /api/auth/register is wrapped in try/catch so a referral failure never blocks account creation.

7) End-to-End Flow Examples

Free event commit:
1. User taps "I'm in" on event detail page.
2. UI calls POST /api/events/:id/commit.
3. Server validates JWT, checks capacity, creates EventAttendee in a transaction, increments currentAttendees.
4. Optional NotificationSchedule records are created for reminder dispatch.
5. Response: { status: "committed" }.
6. UI updates button state and shows confirmation.

Paid event commit:
1. Same flow as free event. No payment processing online.
2. Event detail shows a note: "Paid event — arrange payment with the host directly."
3. Ticket is created with paymentStatus: "pending"; QR is available after host confirms payment out of band.

Referral signup:
1. User A sends invite to phone +91XXXXXXXX via POST /api/referrals/share.
2. Twilio sends SMS with link: https://app.com/join?ref=ABCD1234.
3. Contact opens link, /join validates code via GET /api/referrals/:code, marks clicked.
4. Contact taps "Create an account", lands on /signup?ref=ABCD1234.
5. OTP flow proceeds; ref is threaded through verify and password steps.
6. POST /api/auth/register creates user, completes referral: marks invitation "registered", creates Connection(accepted), creates Notifications for both users, increments ReferralLink.registrations.
7. Both users are now connected on first login.

Chat message:
1. User opens chat at /chat/:connectionId.
2. Page connects Socket.io client to room `connection:${connectionId}`.
3. User sends message; POST /api/chat/:connectionId creates Message in DB.
4. API emits "new_message" to the room via globalThis.io.emitToRoom.
5. Both participants receive the message in real time without polling.

Notification dispatch (cron):
1. Vercel Cron calls GET /api/notifications/dispatch every 5 minutes with Bearer CRON_SECRET.
2. Handler queries NotificationSchedules where sendAt <= now and sent = false.
3. For each schedule, sends email (Nodemailer) or SMS (Twilio), creates Notification record, marks schedule sent.

10) Security

- SQL injection: Prisma parameterized queries.
- Passwords: bcrypt hashed.
- OTPs: Twilio Verify for phone (server-side), hashed storage for email OTPs.
- Sessions: JWT validated in middleware and API routes.
- Ownership: all mutating endpoints verify the requesting user owns the resource.
- Storage: Supabase service role key used only server-side; client never sees it.
- Cron: dual-auth (NOTIFICATION_DISPATCH_SECRET or Vercel CRON_SECRET Bearer).
- Referral: completion is non-fatal and does not expose user data to the referrer beyond a welcome notification.
- Profile visibility: GET /api/users/:id checks Visibility enum before returning data.

Hardening (recommended):
- Rate limiting per IP/user for OTP and login endpoints.
- Secure HTTP headers (CSP, HSTS, X-Frame-Options, Referrer-Policy) via next.config headers.
- Audit logs for critical actions (deactivate, refund, event delete).

12) Challenges and Solutions

- Realtime chat in App Router: App Router does not support persistent TCP connections.
  Solution: Bootstrap Socket.io in pages/api/socketio.ts (legacy pages handler) and expose a global emitToRoom helper via globalThis.io so app/api chat routes can emit without HTTP round-trips.

- Referral code lost through multi-step signup: The join page linked to /signup without the ref query param, breaking the referral chain.
  Solution: Thread ref through every URL step (join -> signup -> verify -> password) and include it in the register POST body. Completion is inline in the register handler (non-fatal).

- PgBouncer vs direct for migrations: Prisma cannot run schema migrations through PgBouncer transaction pooler (prepared statements are not supported in that mode).
  Solution: prisma.config.ts exposes directUrl: env('DIRECT_URL') pointing to the direct Supabase connection. Runtime queries use DATABASE_URL (pooled); migrations use DIRECT_URL.

- Notification dispatch without a worker: No separate background process available on Vercel.
  Solution: vercel.json cron calls GET /api/notifications/dispatch every 5 minutes. Dual-auth allows both Vercel-signed requests and manual calls in development.

- Contact sync on desktop browsers: navigator.contacts (Web Contact Picker API) is only available on Chrome Android and Safari iOS 16+.
  Solution: Feature-detect at runtime with "contacts" in navigator; show a descriptive fallback message on unsupported environments rather than crashing.

- Event payments without Stripe: Stripe was removed to reduce scope and PCI complexity.
  Solution: Paid events are committed identically to free events. The event detail page shows a "contact the host directly" note. Tickets track paymentStatus for the host to update manually.

14) Interview Questions and Strong Answers

1. Why choose Next.js App Router?
   Unified full-stack routing, SSR/CSR flexibility per route, API colocation, and built-in middleware for auth checks at the edge.

2. How is Socket.io bootstrapped in an App Router project?
   pages/api/socketio.ts is a legacy Next.js handler that supports persistent upgrade connections. The server instance is stored in globalThis.io and accessed from app/api routes via a thin emitToRoom helper in lib/socketServer.ts.

3. How do you prevent duplicate EventAttendee rows?
   A unique index on (eventId, userId) in the database; the Prisma transaction that creates the attendee reads currentAttendees inside the transaction, returns 409 if full, and increments atomically.

4. How do you enforce event capacity under concurrency?
   Prisma transaction wraps the attendee count read and the create; if two requests race, one will fail the capacity check or hit the unique constraint. A 409 is returned to the client.

5. How do you secure OTP flows?
   Email OTPs are hashed before storage. Twilio Verify handles phone OTPs server-side. All tokens have an expiry and a usedAt flag. Resend cooldown is enforced by checking the most recent OTP timestamp.

6. How is chat authorization enforced?
   GET and POST /api/chat/:connectionId query the Connection row to verify the requesting userId is user1Id or user2Id before reading or writing messages.

7. How does the referral flow work end-to-end?
   Invitation is sent via Twilio SMS with a unique referralCode. The /join page validates the code and passes it through the entire signup URL chain. On registration, the register API finds the ContactInvitation by code, updates its status, creates an accepted Connection, notifies both users, and increments ReferralLink stats — all in a non-fatal try/catch so a referral failure never blocks signup.

8. How do you handle event discovery relevance?
   Scoring combines: inverse distance (closer events score higher), preference match (category overlap), time-to-start (events soon score higher), popularity (normalized attendee count), and connection signal (attendees you are connected to).

9. Why use DIRECT_URL and DATABASE_URL separately?
   DATABASE_URL points to the PgBouncer pooler for efficient runtime connection management. DIRECT_URL bypasses the pooler for prisma migrate deploy, which requires persistent connections and prepared statements.

10. How do you scale Socket.io?
    Add a Redis adapter (socket.io-redis) so all Node instances share a pub/sub channel. Combine with sticky sessions or move to a dedicated realtime service.

11. How are images uploaded securely?
    Server-side multipart upload to Supabase Storage using the service role key. The client never touches storage credentials; it only receives the returned public URL.

12. How does notification dispatch work?
    NotificationSchedule rows are created when users commit to events. Vercel Cron calls GET /api/notifications/dispatch every 5 minutes. The handler queries unsent schedules where sendAt <= now, sends email or SMS, creates Notification records, and marks schedules sent.

13. How do you enforce onboarding?
    middleware.ts checks the JWT profileComplete claim. If false and the route is not in the onboarding group, the user is redirected to /onboarding/location.

14. What prevents access by deactivated users?
    JWT includes isDeactivated. Middleware checks it and redirects deactivated users. The claim is refreshed from DB on NextAuth JWT callback.

15. How do you optimize DB queries?
    Limit result sets, use Prisma select to fetch only required fields, rely on indexes for frequent filters (userId, eventId, referralCode, contact), and use PgBouncer for connection pooling.

16. How are refunds handled without Stripe?
    POST /api/tickets/refund validates ownership and a 48-hour cancellation window, sets paymentStatus to "refunded", deletes the EventAttendee, and decrements currentAttendees. The actual money movement is handled out-of-band between host and attendee.

17. How would you add caching?
    Cache the event feed and connection suggestions in Redis with a 1-5 minute TTL keyed by userId+location. Invalidate on new event creation or significant location change.

18. How do you ensure data privacy for user profiles?
    GET /api/users/:id checks the target user's visibility setting: public returns full profile, connections returns profile only if a Connection(accepted) exists between requester and target, private returns 404.

19. How do you prevent race conditions in ticket creation?
    A Prisma transaction reads current capacity and creates the ticket atomically. A unique constraint on qrCode also prevents duplicate tickets.

20. How would you add push notifications?
    Register device tokens at POST /api/notifications/subscribe. Store tokens on the User row. In the dispatch handler, add a Web Push or FCM send alongside the existing email/SMS path. Fall back to in-app Notification row if push fails.


15) Elevator Pitch

OUTTACOUCH is an event-first social platform that uses real attendance to drive meaningful connections.
Users discover nearby events, commit to attend, and automatically connect with people they meet in person.
Hosts create and manage events with tools for attendee tracking, QR check-in, and gallery management.
Friends are invited via phone contacts using a referral system — when they sign up through the link, they are instantly connected to the person who invited them.
The system is built on Next.js 15 with Prisma and PostgreSQL, Supabase for media storage, Twilio for OTP and SMS invitations, Socket.io for realtime chat, and Vercel Cron for notification dispatch.
The focus is on relevance, real-world trust, and a clean user experience that moves people from browsing to actually showing up.
